# VST AIコーディング 手順

最終同期日: 2026-04-20

** この記事をAIに送って解説してもらうのもいいです
	「この記事の内容に従って開発したい。何からしたらいい」って言ってコピペとか

*** 大まかな流れ	
	ブラウザの会話用AIと作りたいものについて話す
	要件が明確になったら、要件定義書を作成
	コーディングAIに投げるための、ステップごとのプロンプトを出力
	コーディングAIに投げて、.vst3がビルドされたら各ステップごとにテスト
	問題があれば会話用AIとトラブルシューティング

**** 0. 準備(必要なもの)
	* 欲しいVSTについてのイメージを固める
		なんとなくでOK
		リファレンスがあるといい
		会話用AIに要件を伝える際、「JUCEを使って開発したい」と宣言するのがおすすめ。
	* ブラウザで話すためのAI(以後、会話用AIと呼びます)
		無料ならClaudeの無料枠でSonnetで話すとか、Qwen Chatとか
		有料のやつ持ってるならそれでも。俺はGeminiの学生枠があるので使いました
	* コーディング用AI
		無料で動かすならAntigravityが一番楽だと思う
		GitHub Copilotの学生枠ならGPT-5.3 Codex(xhigh)がよかった
		環境構築が苦じゃなければClaude Code+openrouterの安いモデルとかもあり
	* Visual Studio、「C++によるデスクトップ開発」ワークロード
		VS 2022(17)でもVS 2026(18)でもうまくいく
			CMake単体も入れとくと便利

**** 1. 要件定義(目的・機能の明確化) +プロンプト作り
	会話用AIと話して、「どんな手段で何を実現したいのか」を明確にする。
	それをもとに、ステップごとのプロンプトを作らせる。
		例えば、https://github.com/e11laa/e1_morph はこんな感じ。俺側の発言をパクってくれていい。
			俺「ボコーダー以外にも、2つの信号を使ったモーフィングの方法はないかな。俺が見たものだと、cepstrumを使ってリシンセサイズするものだったんだけど、それ以外にも方法があれば教えてほしい」
			AI「こんなのありますよ(いくつか選択肢を提示する)」
			俺「最適輸送法ってやつ合いそうだな。オープンソースでこれらを実現してるプロジェクトはある？」
			AI「FluCoMaが発表した実装とか、PythonOTとかありますね」
			俺「最適輸送でvstを開発したい。どんな要件を設定し、どんな方針で作るべき？」
			AI「(技術と開発ステップの話をしている)」
			俺「* これを作るため、バイブコーディング用のプロンプトを細かく設定したい。まずはステップ1から」
			AI「これ送ればいいですよ(コーディング用AIのためのプロンプトを添付する)」
	ステップについては、以下をチェックする。条件を満たしていないなら、会話用AIに修正を伝える。
		5-10ステップぐらい。
		GUIは最後に実装する。中身を作ってから見た目を作る
	会話用AIとまとめた『要件定義』や『現在のステップ』は、「AI_Youken.txt」「Step1.txt」のようなテキストファイルとしてプロジェクトフォルダに保存しておくのも便利。

**** 2. コーディングAIに送る
	今回のプロジェクト用にワークスペースを作成して、エディタのAIにさっきのプロンプト(「コーディング用AIのためのプロンプト」)を送る。
	たぶんビルドまでやってくれるはず。なんかエラー出たら会話用AIにエラー(赤いとこ全部)を送り付ければOK。
		結構変な場所に.vst3ファイルが作られるので検索して探してください(多分プロジェクト名_artefacts/Release/VST3/プロジェクト名.vst3/x86_64-winにあると思う)
			なければビルドの手順を会話AIに聞いてください。
		見つけたら、それをC:\Program Files\Common Files\VST3に移動。これでDAWで読めるはずです。
		もしDAWで読み込めない、または一覧に表示されない場合は、弾かれている可能性があります
			DAWの環境設定から『プラグインの再スキャン』を行うか、再起動してみてください。

**** 3. テスト・修正

	ステップごとにDAWでテストする方がいいです。「デフォルト設定で刺しても音が変わらないか」「音量が勝手に変わらないか」などをテストしましょう。
		*** 耳を保護するため、絶対にマスターにリミッターを刺してください。
		まずは読み込めるかどうか。読みこんでDAWがフリーズしたりしたら、会話用AIに伝えましょう。
		音が勝手に変にされたり、小さくされたりしてたら、アナライザーを使ってその特徴を会話用AIに伝えて、トラブルシューティングをしましょう。
			例えば、「66.2dB下げられてしまう」「高域が削られてしまう」など。正確であればあるほど会話用AIが原因を特定しやすいです。
			アナライザーは無料のSPANがいいと思います。
			現状のソースコードを随時送るのも特定に繋がるかも
	トラブルシューティングの中で、会話AIはコーディングAIが問題を修正するためのプロンプトを投げてくれるはずです(投げてくれなかったら要求しましょう)。

**** 繰り返す
	問題なければ/問題が解決したら、1.に戻り、開発の次のステップに進みましょう。
		会話用AIに「次のステップに行こう」というと多分同様に次のステップのプロンプトを出してくれるはず。
			くれなかったら要求しましょう

多分以上の手順でできるはずです！できなければdiscordとかで連絡してください。

**** 補足(割と読んでほしい)
	gitの導入を強くおすすめします。AIがミスる場合を考えるとバージョンの巻き戻しができるほうが安全です。
	macosについては多分追加の手順が必要です。
		Automatic Reference Countingの有効化とかしたら良かったはずです。mac持ってないので試せません、すみません
	細かいステップごとのプロンプトを投げるのが重要です。
		AIが一発で100%完璧なコードを生成してくれるとは限らないので、『動くもの』を作ってから、少しずつ機能を追加していく考え方が適しています。
	VST3 SDKを使うよりJUCEの方が手軽だし情報も多いのでどっちかといえばおすすめです。
		「JUCEで作りたい」と会話用AIに伝えるのも手です。
 ソースコード(Sourceフォルダの中身)を添付したうえで、「エラーメッセージ＋期待動作＋実際の動作」の3点を伝えると、AIが効果的に修正プロンプトを生成しやすくなります。
 	ソースコードの添付については、githubに上げてあるとAIにリンクを送ったりするだけでいいので楽です。
 	C++プロジェクトには主に「宣言を書くファイル(.h)」と「中身を書くファイル(.cpp)」の2つが存在します。エラーが起きたときは、片方だけではなく関連するファイル両方をAIに渡すと、AIが辻褄を合わせやすくなります。
 VSTプラグインのprocessBlock()はリアルタイムオーディオスレッドで実行されるため、メモリ確保やファイル入出力などのブロッキング処理を禁じているので注意です。例えば:
 	 new / malloc などの動的メモリ確保
 	 std::mutex などのロック処理
 	 ファイル読み書きやネットワーク通信
 	会話用AIにあらかじめ注意喚起しておくのもいいですね
 VST3はビルドが結構鬼門なのでがんばってください
 	ビルドエラーが出た際、Visual Studio等の「出力（Output）」ウィンドウの下部にあるエラーの赤い文字の周辺、またはエラーコード（C2065など）を含む行を会話用AIにそのままコピーして渡すといいです。わかんなければ最悪ログを全文渡してもいいと思う。
 2-3ステップごとに、会話を要約してリフレッシュしたり、文脈を思い出させたりするのも手です。
 	会話が長く続くとAIも「何の話だっけ？」ってなっちゃうので。
